## 2) Section 5 - JPA

### 1. JPA의 필요성

- SQL 문자열을 직접 작성하는 것은 다음과 같은 이유 때문에 비효율적임
    1. SQL 명령 작성 시 실수로 인한 런타임 오류 - 컴파일 타임에 오류 인지 힘듦
    → 자바스크립트 대신 타입스크립트 쓰는 그런 느낌적인 느낌
    2. 특정 DB에 종속
    3. 반복 작업 과다
    4. DB의 테이블과 객체 간 괴리 → Django ORM 생각하면 됨

### 2. JPA 이해하기(DjangoORM - SpringBoot JPA)

- JPA는 결론적으로 자바 / 스프링부트 프레임워크의 ORM 기술 표준임
- Django 에서 Django ORM을 이용하듯 스프링부트에서는 JPA를 이용한다~ 정도로 이해하고 우선 넘어가자
    - ORM을 쉽게 설명하면 DB의 테이블과 백엔드 프레임워크의 괴리를 매워주는 도구

### 3. JPA와 `영속성 컨텍스트`

- JPA의 본질적 이해를 위해서는 반드시 영속성 컨텍스트를 이해해야함.
- JPA의 작동 흐름
    - 영속성 컨텍스트(1차 캐시)에 올림
    - 객체 상태 변경 추적
    - 트랜잭션 종료 시점 SQL문 실행

### 4. Hibernate

- Hibernate : JPA 프레임워크 중 하나. 내부적으로 JDBC를 활용함

### `5. 트랜잭션`

- 트랜잭션이란? DB의 상태를 하나의 논리적인 단위로 묶은 실행 단위
    
    → JPA라는 ORM 동작의 기준 단위
    
- **`트랜잭션 이해하기`**
    
    <aside>
    
    A 계좌에서 B 계좌로 10만원을 송금한다.
    
    송금이라는 하나의 행위지만 DB 관점에서는 두 가지 변화가 필요하다.
    
    1) A 계좌 -10만원
    2) B 계좌 +10만원
    
    만약 A 계좌에서 10만원을 빼는 것은 DB 조작에 성공했지만,
    
    B 계좌에 10만원을 더하는 DB 조작에는 실패한다면??
    
    10만원이 그냥 증발한 것과 같다.
    
    이런 데이터 정합성 문제의 발생을 예방하기 위해서는
    
    A 계좌에서 10만원을 빼는 것과 B 계좌에 10만원을 더하는 것을 하나의 트랜잭션으로 묶고
    
    둘 중 하나라도 실패한다면? Rollback 하는 것이 트랜잭션의 기본 개념
    
    </aside>
    
- `트랜잭션의 ACID 특성`
    - Atomicity : 원자성 → 다 성공하거나 다 싪패
    - consistency : 일관성 → 트랜잭션 전/후로 규칙은 유지
    - Isolation : 격리성 → 트랜잭션간 간섭 X
    - Durability : 지속성 → Commit 된 데이터는 영구 저장됨
    
- `Django에서의 트랜잭션`
    - Django는 기본적으로 트랜잭션을 프레임워크에서 자동 관리한다.
    - Django ORM은 DB / 쿼리 중심 프레임워크로 쿼리셋에 따라 SQL 명령이 즉시 실행되기에 상대적으로 트랜잭션의 중요도가 낮다.
    - JPA는 객체 상태 중심의 트랜잭션 기반 상태 관리 시스템이기에 트랜잭션 종료 시점에 SQL 명령이 실행되므로 영속성컨텍스트 - 트랜잭션이 매우매우 중요함!
