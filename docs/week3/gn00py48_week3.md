## 배포

### 1. 배포란 무엇인가
- 배포란 최종 사용자에게 SW를 전달하는 과정이다 오?
- 프로젝트를 외부 사용자가 접속할 수 있는 전용 컴퓨터에 옮겨서 실행하는 것 = 배포

### 2. 배포 왜 함?
- 내 컴퓨터는 24시간 켜져있지 안호, 최종 사용자가 접속하기에 부적절하기 때문에, 개발용 컴퓨터와 서비스용 컴퓨터를 분리하는 작업을 위해 필요하다.

### 3. 배포의 본질

- 내 컴퓨터 = 개발 환경
- 전용 컴퓨터 = 서비스 운영 환경
- 배포 = 개발 환경에서 만든 서버를 운영 환경으로 옮겨 실행하는 것

-> 단순히 “코드 올리기”가 아니라 
코드
실행 환경
DB 연결 정보
서버 설정
를 포함해서 서비스가 실제로 동작하도록 만드는 것이 핵심적인 과정이다.

### 4. 전용 컴퓨터가 왜 필요해?

사용자 입장에서는 서버가 
1. 언제나 켜져 있어야 하고
2. 빠르게 응답해야 하며
3. 일정하게 동작해야 한다.
하지만, 현실적으로 개인이 하기 힘들어서, AWS에서 사서 쓰는거임.

### 5. AWS와 EC2

- AWS(Amazon Web Service)는 아마존이 제공하는 클라우드 서비스이다. 
- EC2는 AWS에서 제공하는 가상 서버 컴퓨터이다.

즉, EC2 = 배포하려고 빌리는 리눅스 서버

## profile과 H2 DB



### 1. profile의 필요성

- 같은 코드라도 실행 환경에 따라 설정은 달라져야 한다.
- 예를 들면

- 내 컴퓨터에서는 내 DB 사용
- EC2에서는 EC2 DB 사용

→ 즉, 코드는 같고 설정만 달라야 하는 상황이 생긴다.

- 이걸 해결하는 개념이 profile이다.



### 2. profile이란 무엇인가

- profile은 --실행 환경별 설정 묶음--이다.

- 보통 이렇게 나눈다.

```text
local = 개발 환경
dev = 배포 환경
default = 기본값
```



### 3. 왜 필요함?

- profile이 없으면

  - DB 설정
  - 서버 설정
    을 코드에서 직접 바꿔야 한다.

- 그러면

  - 실수 발생
  - 환경 전환 번거로움
  - 협업 시 혼선

→ 그래서
코드 수정 없이 환경만 바꾸기 위해 필요함



### 4. local / dev 예시

- local → H2 사용
- dev → MySQL 사용

```text
같은 코드
+ local 실행 → H2
+ dev 실행 → MySQL
```



### 5. H2 DB란 무엇인가

- H2는 경량 데이터베이스
- 메모리에 저장되는 DB



### 6. H2를 왜 씀?

- 서버 꺼지면 데이터 날아감

→ 단점 같지만 개발에서는 장점

- 테이블 자주 바뀜
- 데이터 중요하지 않음
- 빠르게 테스트 가능



### 7. application.yml에서 profile 분리

- profile마다 설정을 따로 둔다

```yaml
local:
  DB = H2
  ddl-auto = create

dev:
  DB = MySQL
  ddl-auto = none
```



### 8. local에서 create 쓰는 이유

- H2는 어차피 데이터 사라짐
- 매번 테이블 새로 만들어도 문제 없음

→ DB 관리보다 개발에 집중 가능



### 9. dev에서 MySQL 쓰는 이유

- 배포 환경에서는 데이터 유지 필요
- 메모리 DB는 불가능

→ 디스크 기반 DB 필요 (MySQL)



### 10. profile 정리

- 코드 수정 없이 환경 변경 가능
- 개발 / 배포 분리 가능
- 실무에서는 local / dev / prod로 확장

# 🌸추가 : Docker 🌸

## 1. Docker란 무엇인가

- Docker는 --실행 환경까지 같이 묶어서 배포하는 도구

→ 코드만 보내지 말고, 환경까지 같이 보내서 모두가 같은 환경에서 개발, 테스트 할 수 있게 해준다.



## 2. 왜 필요함?

문제:

- 내 컴퓨터에서는 됨
- 서버에서는 안됨

이유:

- OS 다름
- Java 버전 다름
- 라이브러리 다름

→ 환경 차이 때문



### 해결

- 환경을 이미지로 만들어서 배포



## 3. 핵심 개념

### 1) Image

- 실행 환경 묶음

### 2) Container

- 실행된 상태

### 3) Dockerfile

- 이미지 만드는 설정 파일



## 4. EC2 방식과 공통점

- 목표는 동일

```text
사용자에게 서버 제공
외부 접속 가능하게 만들기
```

공통 과정:

1. 코드 준비
2. 서버 준비
3. 실행 환경 준비
4. 서버 실행
5. 포트 연결



## 5. EC2 vs Docker 차이

### EC2 직접 배포

- 서버 들어가서 직접 설정

```text
Java 설치
DB 설치
git clone
빌드
실행
```


### Docker 배포

- 환경을 이미지로 만들어 실행

```text
Dockerfile 작성
이미지 생성
컨테이너 실행
```


### 핵심 차이

- EC2 → 환경을 서버에서 만듦
- Docker → 환경을 미리 만들어서 가져옴
